2. Visão Geral do Sistema

O Portal de Registro de Matrículas é um sistema web para registro e acompanhamento de matrículas operadas pela equipe Closers da Uniasselvi, contemplando consultores internos (EFETIVO) e externos (HAG).

2.1. Quem utiliza o sistema

  • Consultores — preenchem o formulário de registro a cada matrícula realizada.
  • Gestores e Analistas — acessam o dashboard para acompanhar indicadores, auditar e alterar status.

2.2. Composição do sistema

O sistema é composto por três arquivos HTML independentes, hospedados no GitHub Pages. Não há servidor próprio — toda a lógica roda no navegador do usuário e os dados ficam armazenados no Supabase (banco PostgreSQL gerenciado na nuvem).

4.2. Regras de negócio do formulário

  • Canal derivado automaticamente: ao selecionar o consultor, o sistema identifica se é Interno (EFETIVO) ou HAG com base no domínio do e-mail cadastrado. Um badge colorido é exibido abaixo do campo.
  • Verificação de duplicata: ao sair do campo Número do Contrato, o sistema verifica se o contrato já foi registrado no banco.
  • Regra do dia 10: todo boleto vence no dia 10 de cada mês. Pago antes = Antecipado. Pago após = Atrasado (ajuste feito pelo analista no dashboard).
  • Polo e Consultor obrigatórios via lista: digitação livre não é aceita — o usuário deve selecionar da lista de autocomplete.

4.3. Fluxo de envio

  1. O JavaScript coleta os valores dos campos do formulário.
  2. Deriva automaticamente o canal de atendimento (Interno / Hag) a partir do consultor selecionado.
  3. Verifica duplicata de contrato no Supabase antes de salvar.
  4. Envia via requisição POST para a API REST do Supabase.
  5. Se bem-sucedido: exibe mensagem de sucesso e limpa o formulário.
  6. Se erro: exibe mensagem de erro com detalhes do problema.

4.4. Cadastrar Novo Colaborador

O formulário possui um modal de cadastro de colaborador (protegido pela mesma senha do dashboard). Ao cadastrar, o colaborador é salvo na tabela colaboradores no Supabase e fica disponível imediatamente no autocomplete do formulário sem necessidade de atualização do código.

Nota: Para que o colaborador seja classificado como Interno ou HAG, o sistema verifica o domínio do e-mail cadastrado. Domínios @uniasselvi, @unicesumar e @vitru = EFETIVO (Interno). Qualquer outro domínio = HAG.

6. Área do Consultor (consultor.html)

Painel de desempenho individual para cada consultor da equipe Closers. Permite que o consultor acompanhe somente os próprios resultados, sem visualizar dados de outros colegas. A página não tem senha — o acesso é aberto, mas os dados exibidos são filtrados pelo nome selecionado.

6.1. Como acessar

  1. No formulário principal (index.html), clicar no botão ⭐ Área do Consultor no canto superior direito.
  2. Na página que abrir, digitar o nome no campo de busca e selecionar na lista.
  3. Um badge aparece indicando automaticamente se é Atendimento Interno ou Atendimento HAG.
  4. Clicar em Ver meu desempenho →.
  5. O dashboard carrega com os dados exclusivos do consultor selecionado.
Nota: Para trocar de consultor sem sair da página, usar o botão "↩ Trocar consultor" que aparece na barra de filtros após carregar o dashboard.

6.2. Isolamento de dados

Os dados de consultores Internos (EFETIVO) e HAG são completamente isolados entre si. O sistema filtra os registros por nome do consultor e canal de atendimento simultaneamente, garantindo que um consultor interno não veja registros HAG e vice-versa.

9. Manutenção do Sistema

9.1. Como abrir o projeto no Claude Code

Todas as tarefas de manutenção descritas nesta seção são realizadas com o auxílio do Claude Code — uma ferramenta de IA que lê o projeto e faz as alterações para você, sem precisar entender código. Para que o Claude funcione corretamente, ele precisa ser aberto dentro da pasta certa do projeto.

Passo a passo para abrir o projeto:

  1. Abra o Visual Studio Code (VS Code) no seu computador.
  2. No menu superior, clique em Arquivo → Abrir Pasta.
  3. Navegue até a pasta do projeto:
    OneDrive → Documentos → Projetos_BI → Projetos_BI_Uniassalvi → Formulario_Closers_Uniasselvi
  4. Clique em Selecionar Pasta.
  5. Com a pasta aberta, pressione as teclas Ctrl + Shift + P ao mesmo tempo.
  6. Na caixa que aparecer, digite Claude e selecione a opção "Claude: Open Claude Code" (ou clique no ícone do Claude na barra lateral esquerda).
  7. O Claude Code abrirá um painel de conversa. Ele lê automaticamente o arquivo CLAUDE.md que está na pasta — esse arquivo contém todo o contexto do projeto, então o Claude já sabe como o sistema funciona.
  8. Basta digitar o que você precisa em português e enviar. Exemplos:
    • "Preciso adicionar um novo consultor chamado João Silva com e-mail joao@vitru.com.br"
    • "Preciso alterar a senha do sistema"
    • "Recebi uma nova base de colaboradores, como atualizo?"
Nota: O Claude Code lê o arquivo CLAUDE.md automaticamente sempre que a pasta do projeto está aberta. Esse arquivo funciona como um "manual técnico interno" que explica toda a estrutura do sistema para a IA — por isso não deve ser apagado ou movido.
Importante: Abra sempre a pasta Formulario_Closers_Uniasselvi diretamente, não uma pasta pai (como Projetos_BI). Se outra pasta estiver aberta, o Claude não vai carregar o contexto correto do projeto.

9.2. Alterar a senha do sistema

  1. Abrir o projeto no Claude Code (ver seção 9.1).
  2. Informar: "preciso alterar a senha para [nova senha]".
  3. O Claude gera o hash SHA-256 da nova senha e substitui a constante SENHA_HASH no index.html automaticamente.
  4. Publicar no GitHub Pages (ver seção 10).
Nota: A senha nunca aparece em texto puro no código — apenas o hash SHA-256 é armazenado. Não é necessário editar o código manualmente.

9.3. Adicionar ou atualizar colaboradores em massa

  1. Receber o arquivo Base_Colaboradores_Uniasselvi.xlsx atualizado.
  2. Abrir o projeto no Claude Code (ver seção 9.1).
  3. Informar: "atualizei a base de colaboradores" e indicar o caminho do arquivo.
  4. O Claude lê o Excel, identifica novos colaboradores, classifica pelo domínio do e-mail e atualiza o código automaticamente.
  5. Publicar no GitHub Pages (ver seção 10).

9.4. Adicionar gestores ao formulário

  1. Abrir o projeto no Claude Code (ver seção 9.1).
  2. Informar: "preciso adicionar o gestor [Nome Completo] ao formulário".
  3. O Claude localiza e atualiza o campo automaticamente.
  4. Publicar no GitHub Pages (ver seção 10).

9.5. Corrigir status de registros em massa via SQL

Acessar o SQL Editor no Supabase e executar uma query UPDATE. Exemplo:

UPDATE matriculas
SET status = 'Pago na data do vencimento'
WHERE consultor = 'Nome do Consultor'
  AND data_atendimento >= '2026-06-01';

9.6. Adicionar nova origem ao formulário

  1. Abrir index.html e localizar <select id="origemMatricula">.
  2. Adicionar a opção desejada.
  3. Abrir dashboard.html e atualizar: o select do filtro de Origem, o select inline da tabela e o array de dados do gráfico de rosca.
  4. Salvar e publicar.

11.2. Como transferir o acesso ao GitHub

A conta GitHub vitrucontactSales é onde o código do sistema está hospedado. Existem duas formas de passar o controle para o novo responsável — escolha a que for mais fácil:

Opção A — Adicionar o novo responsável como colaborador (mais simples)

O novo responsável cria uma conta pessoal no GitHub e é adicionado ao repositório. Ele poderá publicar atualizações normalmente. A URL pública do sistema não muda.

  1. Novo responsável acessa github.com e cria uma conta gratuita com o e-mail corporativo dele.
  2. Responsável atual (ou quem tiver acesso) acessa github.com e faz login com kleberson.azevedo@vitru.com.br / Cri@tur@1.
  3. Ir no repositório: vitrucontactSales / closers-forms-uniasselvi.
  4. Clicar em Settings (aba no topo do repositório).
  5. No menu à esquerda, clicar em Collaborators.
  6. Clicar em Add people e digitar o nome de usuário ou e-mail do novo responsável.
  7. Selecionar permissão Admin e confirmar.
  8. O novo responsável receberá um convite por e-mail — ele precisa aceitar para ter acesso.
Vantagem: não precisa alterar senha nem e-mail da conta atual. A URL do sistema continua a mesma. Indicado se o responsável será alguém interno com e-mail corporativo próprio.

Opção B — Alterar o e-mail da conta vitrucontactSales

A mesma conta GitHub passa a ser acessada pelo novo responsável, com o e-mail e senha dele. Nenhuma URL muda.

  1. Novo responsável acessa github.com e faz login com kleberson.azevedo@vitru.com.br / Cri@tur@1.
  2. Clicar na foto do perfil (canto superior direito) → Settings.
  3. No menu à esquerda, clicar em Emails.
  4. Em "Add email address", digitar o e-mail do novo responsável e clicar em Add.
  5. O GitHub enviará um e-mail de verificação — o novo responsável deve clicar no link recebido.
  6. Após verificar, clicar em Primary ao lado do novo e-mail para torná-lo principal.
  7. Remover o e-mail antigo (kleberson.azevedo@vitru.com.br) clicando em Remove ao lado dele.
  8. No menu à esquerda, clicar em Password and authentication e definir uma nova senha.
Vantagem: o novo responsável passa a ser o único dono da conta. Indicado quando se quer uma transição completa, sem dependência da conta anterior.
Atenção: Na Opção B, realize a troca de e-mail ANTES de Kleberson sair da empresa — é necessário ter acesso ao e-mail antigo para confirmar a remoção. Se o e-mail corporativo já foi desativado, use a Opção A.

11.3. O que NÃO precisa ser transferido

  • Dados do banco: ficam no Supabase na nuvem — acessíveis por qualquer máquina com as credenciais.
  • Código em produção: está no GitHub Pages — acessível pela URL pública.
  • Configurações do sistema: documentadas neste POP e no arquivo CLAUDE.md do projeto.

12. Limitações e Pontos de Atenção

Autenticação simples: A proteção por senha é feita via JavaScript no front-end com hash SHA-256. Um usuário com conhecimento técnico pode contornar a verificação manipulando o sessionStorage diretamente no navegador. Para uso interno com equipe conhecida, o nível atual é adequado. Se necessário maior segurança, considerar implementar autenticação real via Supabase Auth.
Código visível no navegador: Como toda aplicação web estática, o código HTML/JS pode ser visualizado no "Inspecionar" do Chrome. A anon key do Supabase fica visível — isso é esperado e seguro, pois as permissões reais são controladas pelas políticas RLS no banco.
Performance com grandes volumes: O painel carrega todos os registros do banco para o navegador. Se o volume crescer significativamente, pode ser necessário implementar filtros no servidor (query parameters na API do Supabase).
Lista de consultores hardcoded: A lista principal de 159 consultores está no código do index.html. Novos colaboradores cadastrados pelo modal ficam disponíveis imediatamente via Supabase, mas para atualizações em massa é necessário atualizar o código e publicar no GitHub.
Sem notificações automáticas: O sistema não envia e-mails ou alertas. Toda consulta é ativa — o gestor precisa abrir o painel para ver os dados. A funcionalidade de e-mail ao cadastrar colaborador está listada como pendente de implementação.
